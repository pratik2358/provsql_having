\section{Implementation and Experiments}
\label{sec:implementation_experiments}
\pierre{2 pages}

We implemented our semantics in \textsc{ProvSQL} and evaluated 
it on a probabilistic image-retrieval database. Following the method 
in~\cite{yunus2025using}, the database is built from object-detection 
output over the COCO 2017~\cite{lin2014microsoft} dataset: for each 
image, YOLOv5~\cite{ultralytics2021yolov5} detects objects, 
and each detected object is stored as one tuple in relation \sql{dataset} with 
a unique identifier \sql{id}, the image identifier \sql{img}, the object 
class \sql{obj}, and a probability derived from the detector 
confidence score. These object annotations are treated as independent 
probabilistic facts. The database contains $119,359$ images and $579,123$ 
object annotations.

\smallskip\noindent
We compare execution times in ProvSQL for: (i) provenance tracking and, (ii) probability computation 
of a \sql{HAVING} query with comparisons of the form \sql{COUNT(*) >= k} that uses our possible-world semantics against an equivalent self-\sql{JOIN} version that defaults to the \sql{JOIN} semantics.
In our experiments we vary the scalar $k$ of the queries from 1 to 10.
The query along with its \sql{JOIN}-rewite is presented in the Appendix~\ref{sec:appendix_impl_experiments}.

\smallskip\noindent
Figure~\ref{fig:prov_only_times} reports query execution times for provenance evaluation. We test our 
possible-world semantics with the \sql{HAVING} version of the query. In its execution a provenance circuit is created which introduces a \emph{comparison gate} for the aggregate comparison and later a  call to the \texttt{sr\_boolexpr} semiring UDF on the circuit triggers our possible-world semantics which we implemented at a semiring-level. For smaller values of $k$, provenance evaluation using our possible-world semantics is slower 
than that of the \sql{JOIN} version. However with increasing $k$, provenance evaluation using possible world-semantics is considerably faster. This is expected, since the rewriting requires 
$k$ aliases of the base relation and produces increasingly expensive intermediate 
join tables. In comparison, execution time(s) for provenance evaluation using our possible-world semantics is almost constant with varying $k$ in the query.

\smallskip\noindent
We observe similar result in Figure~\ref{fig:prob_comp_times} for probability computation, except that, execution times for probability computation using possible-world semantics is linear with varying $k$. For the equivalent \sql{JOIN} query, performance is unpredictable: linear till $k=6$ and exponential thereafter.\rjok{wait for experiments to finish}.

\subsection{Experiment Codes}

\begin{minted}[fontsize=\small, breaklines, linenos]{sql}
SELECT img, sr_boolexpr(provenance()) AS prov
FROM dataset
WHERE obj = c
GROUP BY img
HAVING COUNT(obj) >= k;
\end{minted}

For probability evaluation, we use use \sql{probability_evaluate(provenance())} 
instead of \sql{sr_boolexpr(provenance())}.

The equivalent self-\sql{JOIN} rewriting used in the comparison is:

\begin{minted}[fontsize=\small, breaklines, linenos]{sql}
SELECT img, provenance()
FROM (
  SELECT DISTINCT d1.img
  FROM dataset d1
  JOIN dataset d2
    ON d2.img = d1.img AND d2.obj = c AND d2.id > d1.id
  ...
  JOIN dataset dk
    ON dk.img = d1.img AND dk.obj = c AND dk.id > d{k-1}.id
  WHERE d1.obj = c
) AS t;
\end{minted}
\end{toappendix} 
We use the same probabilistic backend (D4) in ProvSQL, for both the queries. For each $k$, every query was executed $20$ times on a 12th Gen Intel(R) Core(TM) i7-12700 machine running Ubuntu 22.04 using ProvSQL as an extension to Postgres 18 with shared buffer size limited to 128MB.
