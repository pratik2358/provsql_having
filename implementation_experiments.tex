\section{Implementation and Experiments}
\label{sec:implementation_experiments}
\subsection{Implementation}
We implemented the algorithms described in Section~\ref{sec:algorithms}
(Proposition~\ref{prop:algorithms})
within ProvSQL in C++. Our implementation includes a number of optimizations. One such optimization is a
generic \emph{range check}: for every aggregation operator, we determine
what the minimum and maximum value it can take over all possible worlds (e.g., the sum of all
negative numbers and of all positive numbers respectively for \sql{SUM})
and return either $\mathbb{0}$ if a condition is unsatisfiable, or the
$\oplus$ of all possible worlds if a condition is necessarily
true.

Similarly, for \sql{SUM}, instead of running the dynamic programming up
to the comparison constant $C$, the maximum tracked sum $J$ is defined in
an optimized way (cf., lines 8--13 in Algorithm~\ref{alg:sum_dp}, for the
case of non-negative integers).
We also eliminate iterations over sums that are not yet reachable at a
particular DP-iteration, reducing early-stage DP cost. We exploit when
possible the absorptive nature of the m-semiring.

For probability evaluation, we also implemented the algorithms presented
in Proposition~\ref{prop:agg-cmp-poly-prob}.

Finally, while constructing semiring provenance expressions, we fold standard
(m-)semiring identities: multiplication by $\mathbb 1$, addition or
multiplication by $\mathbb 0$, etc.

\subsection{Experiments}
To test our implementation and validate the practical feasibility of the
\sql{HAVING} semantics, we ran experiments on a probabilistic image-retrieval database. Following the method
in~\cite{yunus2025using}, the database is built from object-detection
output over the COCO 2017~\cite{lin2014microsoft} dataset: for each
image, YOLOv5~\cite{ultralytics2021yolov5} detects objects,
and each detected object is stored as one tuple in relation \sql{dataset} with
a unique identifier \sql{id}, the image identifier \sql{img}, the object
class \sql{obj}, and a probability derived from the detector
confidence score. These object annotations are treated as independent
probabilistic facts. The database contains $119{,}359$ images and $579{,}123$
object annotations.

We compare execution times in ProvSQL for: (i)~provenance tracking (with
production of a symbolic formula representing the provenance); and
(ii)~probability computation of a \sql{HAVING} query with comparisons of
the form \sql{COUNT(*) >= k} or \sql{COUNT(*) <= k} to find images with
at least (or at most) $k$ objects for a given class, for our
possible-world semantics against the equivalent self-\sql{JOIN} version
(see Theorem~\ref{th:correctness}).
Note that this is the exact set-up of \cite{yunus2025using}, where
authors had to use the \sql{JOIN} version as nothing else was available.

We choose a specific object class (72, \emph{refrigerator}).
In our experiments we vary the scalar $k$ of the queries from 0 to 15
(the maximum number of refrigerators detected in an image is 13).
The exact queries we ran, along with their \sql{JOIN}-rewrites, are presented in
the Appendix~\ref{sec:appendix_impl_experiments}. For probabilistic query
evaluation, we use the default ProvSQL pipeline which, when the
provenance circuit is not read-once, first attempts to build a
tree-decomposition of small treewidth and, if unsuccessful, resorts to an
external knowledge compiler (d4~\cite{DBLP:conf/ijcai/LagniezM17}).

For each $k$, every query was executed $3$ times on a standard laptop
computer running Ubuntu 26.04, ProvSQL version 1.8.0 on top of PostgreSQL 18.

Figure~\ref{fig:prov_only_times_ge} and Figure~\ref{fig:prov_only_times_lt} report query
execution times including the materialization of a textual representation
of a provenance formula, comparing our possible-world semantics
with the corresponding \sql{JOIN}-based rewriting. During execution, our semantics constructs
a provenance circuit with a \emph{comparison gate} representing the aggregate comparison. The
possible-world semantics is triggered at the semiring evaluation level,
by instantiating the formula pseudo-semiring (\texttt{sr\_semiring})
in ProvSQL. For provenance computation alone, the running time of our semantics remains relatively
stable as $k$ varies. For the ``\sql{COUNT(*) >= k}'' query, the \sql{JOIN}-based rewriting is faster for
very small values of $k$, and then becomes slower than the \sql{HAVING}-based semantics after a
phase shift around $k=6$ -- queries with a large number of joins can be
slow for the query evaluator. For the ``\sql{COUNT(*) <= k}'' query, the
\sql{JOIN}-based rewriting (which includes a non-monotone \sql{EXCEPT}) is again
cheaper for very small values of $k$, but becomes several orders of
magnitude more expensive than our semantics for larger
values of $k$. Thus, for provenance computation alone, the main
observation is the stability of the runtime cost, which pays off as $k$
increases.

The difference becomes much more pronounced in Figure~\ref{fig:prob_eval_times_ge} and
Figure~\ref{fig:prob_eval_times_lt}, where probability evaluation is included in addition
to query execution and provenance computation. In these experiments,
we compare three different probability computation modes: \sql{HAVING}
semantics with the Poisson-binomial CDF optimization for counting independent events
of Proposition~\ref{prop:agg-cmp-poly-prob}, plain \sql{HAVING} semantics, and
\sql{JOIN} rewriting; recall that probability evaluation is, in
general, \#P-hard. The first one remains efficient for all values of~$k$;
the second one is more efficient than the \sql{JOIN} semantics for the
monotone $\geq k$ case, but is an order of magnitude slower for the $\leq k$
case, where the non-monotone and larger provenance expressions become
hard to process for the knowledge compiler. The \sql{JOIN} rewriting
presents a similar profile in both cases (note the different scale of
both graphs) -- note it is unable to exploit the fact that
the query becomes vacuously true or false when $k$ is higher than the
number of objects in the images, which both our implementations can.

\begin{toappendix}
  \section{Material for Section~\ref{sec:implementation_experiments}
  (Implementation and Experiments)}
\newcommand{\quotedzero}{\PYG{l+s+s1}{\apos$\mathbb{0}$\apos}}
\label{sec:appendix_impl_experiments}

We present the query templates used for the experiments of
Section~\ref{sec:implementation_experiments}. They are SQL queries using
the functions of the ProvSQL extension of PostgreSQL (\sql{provenance()}, \sql{sr_formula},
\sql{probability_evaluate}, etc.)

The \sql{HAVING}-based template, instantiated either with
``\sql{COUNT(*) >= k}'' or ``\sql{COUNT(*) <= k}'', is:
\newcommand{\apos}{\texttt{\textquotesingle}}
\begin{minted}[fontsize=\small, breaklines, escapeinside=||]{postgresql}
SELECT * FROM (
  SELECT img, sr_formula(provenance(), 'image_mapping') AS prov
  FROM dataset
  WHERE obj = c
  GROUP BY img
  HAVING COUNT(*) >= k  -- or:  HAVING COUNT(*) <= k
) sub WHERE prov <> |\quotedzero|;
\end{minted}

For probability evaluation, the projection
\sql{sr_formula(provenance(), 'image_mapping')} is replaced by
\sql{probability_evaluate(provenance())} and the zero-filter
``\mintinline[escapeinside=||]{postgresql}{prov <> |\quotedzero|}'' by
``\sql{prov > 0}''; both serve to discard images whose
provenance simplifies to $\mathbb{0}$ (resp.\ whose probability evaluates to probability~$0$), which
under the \sql{HAVING} semantics correspond to images for which the
aggregate comparison is impossible in every possible world.

The equivalent self-\sql{JOIN} rewriting for the ``\sql{>= k}'' case uses
a $(k{-}1)$-fold self-join, with the
``\sql{id > }'' chain ensuring distinct matching tuples:
\begin{minted}[fontsize=\small, breaklines, escapeinside=||]{postgresql}
SELECT * FROM (
  SELECT img, sr_formula(provenance(), 'image_mapping') AS prov
  FROM (
    SELECT DISTINCT d1.img
    FROM dataset d1
    JOIN dataset d2 ON d2.img = d1.img AND d2.obj = c AND d2.id > d1.id
    ...
    JOIN dataset dk ON dk.img = d1.img AND dk.obj = c AND dk.id > d|\(_{k-1}\)|.id
    WHERE d1.obj = c
  ) AS t
) sub WHERE prov <> |\quotedzero|;
\end{minted}

The ``\sql{<= k}'' case is non-monotone and is rewritten as
“at least one match”~\sql{EXCEPT}~“at least~$k+1$ matches”:
\begin{minted}[fontsize=\small, breaklines, escapeinside=||]{postgresql}
SELECT * FROM (
  SELECT img, sr_formula(provenance(), 'image_mapping') AS prov
  FROM (
    SELECT DISTINCT d1.img
    FROM dataset d1
    WHERE d1.obj = c
    EXCEPT
    SELECT DISTINCT d1.img
    FROM dataset d1
    JOIN dataset d2 ON d2.img = d1.img AND d2.obj = c AND d2.id > d1.id
    ...
    JOIN dataset d|\(_{k+1}\)| ON d|\(_{k+1}\)|.img = d1.img AND d|\(_{k+1}\)|.obj = c AND d|\(_{k+1}\)|.id > dk.id
    WHERE d1.obj = c
  ) AS t
) sub WHERE prov <> |\quotedzero|;
\end{minted}
The same projection / zero-filter substitution applies to obtain the
\sql{probability_evaluate} variants of both \sql{JOIN} rewritings.
