\subsection{Implementation}
We implemented the algorithms described in Section~\ref{sec:algorithms} in ProvSQL with some added optimizations for 
improving performance. Our implementation targets \sql{SUM}, \sql{COUNT}, \sql{MIN}, \sql{MAX}, \sql{AVG} and \fakesql{PICKFIRST} with specialized optimizations for \sql{SUM} and \sql{COUNT}.

\smallskip\noindent
\emph{\textbf{Optimizations}} For aggregate comparisons involving \sql{SUM} we deal with impossible-world cases as established in Algorithm~\ref{alg:sum_dp} itself. In a similar fashion the following predicates are detected early on as universally true:
\begin{itemize}
  \item \sql{SUM > C} and $C < 0$
  \item \sql{SUM >= C} and $C \leq 0$
  \item \sql{SUM < C} and $C > T$
  \item \sql{SUM <= C} and $C \geq T$
\end{itemize}
For these cases we skip DP computation and compute fallback to a brute-force enumeration of possible worlds. 
Instead of allocating DP up to the comparison constant $C$, the maximum tracked sum $J$ is defined in an optimized way (lines 8-13 in ALgorithm~\ref{alg:sum_dp}) to prevent unnecessarily large DP tables when $C$ is much larger than reachable sums.
We also eliminate iterations over sums that are not yet reachable at a particular DP-iteration, reducing early-stage DP cost. For negative tuple values, we resort to exhaustive enumeration of possible worlds.
For absorptive semirings if \sql{SUM} in a possible-world already satisfies a monotone comparison ($\ge, >$) , further supersets of the world are not enumerated because under absorption, supersets of satisfying worlds do not contribute new information.

\smallskip\noindent
Similarly, for \sql{COUNT >= m} or \sql{COUNT > m} under absorptive semirings, instead of enumerating all subsets of size at least $m$ only subsets of size exactly $m$ are generated.
Further if every non-empty world satisfies the comparison predicate and the semiring is absorptive, the set of all worlds is replaced with singleton worlds representing an upward closure via minimal worlds.

\smallskip\noindent
\emph{\textbf{Simplifications}} While constructing semiring provenance expressions multiplication by $\mathbb 1$ and addition of $\mathbb 0$ is omitted to reduce clutter.
In absorptive semirings with monotone comparisons for \sql{MAX} with $>\text{ or }\geq$
and for \sql{MIN} with $<\text{ or }\leq$ missing-tuples factor is suppressed when they do not affect minimal satisfying worlds.

\smallskip\noindent
The presented optimizations preserve correctness under possible-world semantics and semiring evaluation while significantly reducing world enumeration times and provenance expression size.


\subsection{Experiments}
To test our implementation we ran experiments on a probabilistic image-retrieval database. Following the method 
in~\cite{yunus2025using}, the database is built from object-detection 
output over the COCO 2017~\cite{lin2014microsoft} dataset: for each 
image, YOLOv5~\cite{ultralytics2021yolov5} detects objects, 
and each detected object is stored as one tuple in relation \sql{dataset} with 
a unique identifier \sql{id}, the image identifier \sql{img}, the object 
class \sql{obj}, and a probability derived from the detector 
confidence score. These object annotations are treated as independent 
probabilistic facts. The database contains $119,359$ images and $579,123$ 
object annotations.

We compare execution times in ProvSQL for: (i) provenance tracking and, (ii) probability computation of a \sql{HAVING} query with comparisons of the form \sql{COUNT(*) >= k} to find images with at least $k$ refrigerators, for our possible-world semantics against the equivalent self-\sql{JOIN} version. %that defaults to the \sql{JOIN} semantics.
In our experiments we vary the scalar $k$ of the queries from 1 to 9.
The result query along with its \sql{JOIN}-rewrite is presented in the Appendix~\ref{sec:appendix_impl_experiments}. We use the same probabilistic backend (D4) in ProvSQL, for both the queries. For each $k$, every query was executed $20$ times on a 12th Gen Intel(R) Core(TM) i7-12700 machine running Ubuntu 22.04 using ProvSQL as an extension to Postgres 18 with shared buffer size limited to 128MB.


Figure~\ref{fig:prov_only_times} reports query execution times for provenance evaluation, comparing our possible-world semantics with the \sql{HAVING} version of the query. 
During the execution a provenance circuit is created having a \emph{comparison gate} representing the aggregate comparison. 
The possible world semantics are trigger at the semiring level, by using a call to the \texttt{sr\_boolexpr} semiring UDF on the circuit. 
For smaller values of $k$, provenance evaluation using our possible-world semantics is slower than that of the \sql{JOIN} version. 
However with increasing $k$, provenance evaluation using possible world-semantics is considerably faster. 
This is expected, since the rewriting requires $k$ aliases of the base relation and produces increasingly expensive intermediate join tables. 
In comparison, execution times for provenance evaluation using our possible-world semantics are almost constant with varying $k$ in the query.

We observe similar result in Figure~\ref{fig:prob_comp_times} for probability computation. 
In this case, execution times for probability computation using possible-world semantics is linear with varying $k$.

\subsection{Experiment Codes}

The original SQL query template is the following:
\begin{minted}[fontsize=\small, breaklines, linenos]{sql}
SELECT img, sr_boolexpr(provenance()) AS prov
FROM dataset
WHERE obj = c
GROUP BY img
HAVING COUNT(obj) >= k;
\end{minted}

For probability evaluation, we use use \sql{probability_evaluate(provenance())} 
instead of \sql{sr_boolexpr(provenance())}.

The equivalent self-\sql{JOIN} rewriting used in the comparison is:

\begin{minted}[fontsize=\small, breaklines, linenos]{sql}
SELECT img, provenance()
FROM (
  SELECT DISTINCT d1.img
  FROM dataset d1
  JOIN dataset d2
    ON d2.img = d1.img AND d2.obj = c AND d2.id > d1.id
  ...
  JOIN dataset dk
    ON dk.img = d1.img AND dk.obj = c AND dk.id > d{k-1}.id
  WHERE d1.obj = c
) AS t;
\end{minted}
